Builder Pattern
1. 빌더 패턴이란?
빌더 패턴은 복잡한 객체를 단계별로 조립(Build)하여 생성하는 패턴이다.
생성자에 많은 파라미터가 있거나, 다양한 옵션 조합을 지원해야 하는 객체를 만들 때 특히 유용하다.

즉,
생성자를 한 번에 호출하는 대신
메서드 체이닝(Select → From → Where → Build) 방식으로
“조립하듯” 객체를 만든다.

2. 왜 사용하는가?
특히 아래와 같은 문제를 해결하기 위해 사용된다.
생성자 파라미터가 너무 많아지는 문제
new User("홍길동", 25, true, false, null, 300, null, "Korea");
→ 읽기 어렵고 유지보수 최악.
선택적 옵션이 많은 객체 생성
HTTP 요청 / SQL 쿼리 / 환경 설정 객체 등은
옵션이 5개~20개로 늘어나는 경우도 많다.
가독성이 좋아야 하는 구조
SQL, HTTP Request, UI 구성 객체 등은
직관적인 “조립 방식”이 훨씬 읽기 좋다.

3. 구현 포인트
기능	설명
private 필드들	SELECT / FROM / WHERE… 값 저장
Select(), From() 등	단계별 조립 메서드
return this	메서드 체이닝 지원
Build()	최종 SQL 문자열 생성

4. 장점 (Advantages)
✔ 1) 가독성이 매우 좋아짐
메서드 체이닝으로 어떤 옵션을 설정했는지 눈에 보인다.
✔ 2) 선택적 옵션 처리에 탁월
WHERE, ORDER BY 같은 옵션을 필요할 때만 붙이면 된다.
✔ 3) 객체 생성 로직이 명확해짐
읽기 쉬운 코드:
.Select("*").From("Users").Where("Age > 20")
✔ 4) 불변 객체(Immutable Object) 만들 때 좋음
Builder로 조립하고, 최종 결과만 수정 불가로 만들 수 있다.

5. 단점 (Disadvantages)
1) 클래스가 추가로 필요
Builder 클래스가 늘어나서 파일 수가 많아질 수 있다.
2) 너무 단순한 객체에는 과함
필드 2~3개 정도라면 그냥 생성자 쓰는 것이 더 깔끔하다.
3) 메서드 체이닝 오용 가능
아무 생각 없이 Builder만 남용하면 Setter 남발 구조가 된다.
